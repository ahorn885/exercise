## The pattern in one paragraph

For each new third-party provider you're applying to, ship a small PR
that (a) registers a slug in `routes/oauth_callbacks.py`, (b) creates a
`routes/<provider>.py` blueprint with a `/<slug>/webhook` route returning
200, (c) wires the blueprint into `app.py` (register + auth-exempt the
webhook endpoint + CSRF-exempt the blueprint), and (d) optionally adds
the slug to `_PROBED_AT_REGISTRATION` in `oauth_callbacks.py` if the
provider's portal HEAD/GET-probes the redirect URI at form-save time.
Then deploy, verify both URLs return 200, paste them into the
provider's form, and stop. Real OAuth exchange and webhook verification
are deferred to the matching phase in the master plan.

## Recipe for a new provider

Replace `<slug>` (lowercase-hyphenated), `<Name>` (display name), and
`<module>` (Python module name, snake_case) throughout.

### 1. Confirm the slug is registered

`routes/oauth_callbacks.py:_PROVIDERS` already has 18 entries. If your
provider isn't there, append a tuple. Order doesn't matter — the file
groups by wave for readability.

```python
('<slug>', '<Name>'),
```

### 2. Create `routes/<module>.py`

```python
"""<Name> integration.

Phase 0: webhook stub only. Returns 200 OK so the URL checks out if the
provider's portal probes it. <Spec-specific notes — e.g., HMAC scheme,
response-time budget, retry policy.> Real verification and dispatch
ship in the matching phase of the master plan.
"""
from flask import Blueprint, jsonify

bp = Blueprint('<module>', __name__, url_prefix='/<slug>')


@bp.route('/webhook', methods=['GET', 'POST'])
def webhook():
    return jsonify(status='ok'), 200
```

For providers that expect a specific response body (e.g., COROS wants
`{"result":"0000","message":"ok"}`), return that instead of the generic
`{"status":"ok"}`. Always 200; always GET+POST.

### 3. Wire it into `app.py`

Three edits, mirroring the COROS pattern at lines 142, 162, 187:

```python
# Import block (near line 142):
from routes.<module> import bp as <module>_bp

# Registration + CSRF exempt (near line 162):
app.register_blueprint(<module>_bp)
csrf.exempt(<module>_bp)  # webhook pushes are server-to-server, no CSRF token

# Auth exempt (near line 187, inside _AUTH_EXEMPT_ENDPOINTS):
'<module>.webhook',
```

### 4. (Optional) Make the redirect URI probe-friendly

If the provider's developer portal validates the OAuth redirect URI by
probing it at form-save (RWGPS does this), add the slug to
`_PROBED_AT_REGISTRATION` in `routes/oauth_callbacks.py`. This flips
the response from 501 to 200 while keeping the same body. Default for
new providers is 501 — only add to the set if you have evidence the
portal probes.

```python
_PROBED_AT_REGISTRATION: frozenset[str] = frozenset({
    'ride-with-gps',
    '<slug>',   # add here
})
```

### 5. Smoke-test, commit, push, merge

```bash
SECRET_KEY=$(python -c "import secrets;print(secrets.token_urlsafe(48))") python -c "
from app import app
c = app.test_client()
for r in [
    c.get('/<slug>/webhook'),
    c.post('/<slug>/webhook', json={}),
    c.get('/auth/<slug>/callback'),
]:
    print(r.status_code, r.get_data(as_text=True)[:80])
"
```

All three should be 200 (or 200/200/501 if you skipped step 4).

Commit message style (see `git log`): `<module>: <one-line>` followed
by a paragraph explaining the spec quirks for the next person to wire
up the real handler. Always include the Claude session URL footer.

PR title: `<module>: webhook stub …`. Merge with the **merge** method
(not squash) so the implementation commit stays visible in `main`'s
history — the merge-commits are the chronological "what landed when"
trail.

## Env-var convention

Default — most providers issue ONE credential pair covering both OAuth
and webhook auth:

```
<PROVIDER>_CLIENT_ID
<PROVIDER>_CLIENT_SECRET
```

Some providers issue separate pairs (RWGPS does; Polar does for the
webhook HMAC secret specifically). When that happens, name them by
function:

```
<PROVIDER>_API_CLIENT_ID         # outgoing API + incoming-webhook key matching
<PROVIDER>_API_CLIENT_SECRET     # HMAC signing key for webhook verification
<PROVIDER>_OAUTH_CLIENT_ID       # used in authorize URL + token exchange
<PROVIDER>_OAUTH_CLIENT_SECRET   # used in token exchange
```

If the webhook auth is HMAC-based, **note in the route-file docstring
which secret is the signing key** — it's the kind of thing that gets
miswired six months later when the real handler ships.

Set on Vercel **Production + Preview**, marked Sensitive. Never commit
to `.env.example` with real values — names only, empty.

## Questions to ask the user (or check in the partner docs) before shipping

These are the per-provider unknowns that the stub itself doesn't
resolve. Collect answers before promoting the stub past Phase 0:

1. **Does the dev portal probe the URLs at form-save?** If yes → step 4
   in the recipe. If unknown → it's safe to add `_PROBED_AT_REGISTRATION`
   speculatively; no downside.
2. **Webhook auth scheme?** Static token in body? Static credentials in
   headers (COROS)? HMAC over body with a secret (RWGPS, Polar)? Signed
   JWT (rare)?
3. **Expected response body and status?** Most want 200 + JSON success
   envelope. COROS specifically wants `{"result":"0000","message":"ok"}`.
   RWGPS wants 200 within 1 second.
4. **Retry policy?** RWGPS has **no retry** — failed deliveries are lost
   forever. COROS retries 2× then batches with the next push. Polar
   auto-deactivates the webhook after 7 days of failures. Wahoo retries
   30m → 4h → 24h → 72h. This shapes how aggressive the eventual handler
   must be about ack-and-defer.
5. **Verification handshake on URL registration?** Some providers (not
   COROS, not RWGPS, but Stripe/Slack-style) POST a challenge token that
   must be echoed back. If yes, the stub needs to recognize and respond,
   not just return 200.
6. **One credential pair or two?** See env-var convention above.

## What's NOT in scope for these PRs

- Real OAuth code-for-token exchange (lives in `<module>_connect.py`,
  Phase 1 of master plan)
- HMAC / signature verification (lives in the promoted webhook handler,
  matching phase)
- Database tables for tokens (`provider_auth` generic table — Phase 1)
- Webhook event recording (`webhook_events` table — Phase 1)
- `/auth/<slug>/callback` doing the real token exchange — stays 501 (or
  200 if probe-friendly) until Phase 1
- Any UI surface (`/<slug>/connect`, `/<slug>/dashboard` — Phase 1+)

If the user asks for anything beyond the stub pair, **stop and refer to
the master plan** at `/root/.claude/plans/will-this-make-changes-peaceful-boot.md`.
Don't extend Phase 0 with one-off integration code — it'll have to be
rewritten when the generic `provider_auth` foundation lands.
