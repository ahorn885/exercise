# Session Handoff — 2026-05-11

**Scope:** Stubbed OAuth provider callback routes so the redirect URIs
registered in the Garmin / Strava / Polar / Wahoo developer-portal
applications resolve to a real endpoint during provider review.

**Merged to `main`:** PR #21, squash commit `96f9b84`.

---

## What happened

1. Confirmed there's no existing OAuth-redirect plumbing in the app. The
   only third-party auth in the codebase is `garth`-based Garmin Connect
   login (`garmin_connect.py`, `routes/garmin.py`), which uses a
   username/password flow with no callback URL.
2. Identified the production Vercel domain as `aidstation-pro.vercel.app`
   (not previously recorded in the repo).
3. Added stub handlers at `/auth/<provider>/callback` for the four
   providers. Each returns `501 Not Implemented` with a plain-text body
   — enough to pass a "does this URL resolve?" verification ping from a
   provider's developer portal.
4. Wired the new blueprint into `app.py` and added all four endpoints to
   `_AUTH_EXEMPT_ENDPOINTS` so providers can hit them without being
   bounced to `/auth/login`.

## What's in the repo now

- `routes/oauth_callbacks.py` — new blueprint, `url_prefix='/auth'`,
  four routes: `garmin`, `strava`, `polar`, `wahoo`. Each accepts GET
  and POST and returns 501.
- `app.py:139` — import of the new blueprint.
- `app.py:159` — `register_blueprint(oauth_callbacks_bp)`.
- `app.py:173-176` — four new entries in `_AUTH_EXEMPT_ENDPOINTS`
  (`oauth_callbacks.garmin`, `.strava`, `.polar`, `.wahoo`).

## ⚠️ What's pending

### Real OAuth flow logic for each provider

The handlers are 501 stubs. Once API access is granted and you have the
per-provider specs, each handler needs the actual exchange:

- Validate `state` against a value stashed in the user's session before
  redirecting them to the provider.
- Exchange the `code` param for an access token + refresh token.
- Persist tokens scoped to `user_id`. Likely needs a new per-user table
  (something like `provider_auth` with `(user_id, provider)` PK,
  `access_token`, `refresh_token`, `expires_at`, `scopes`, `updated_at`).
  Pattern to follow: `garmin_auth` (one row per user, see
  `DATABASE.md` "Garmin" section).
- Background or on-demand token refresh — most of these are short-lived
  OAuth 2.0 access tokens.

Each provider differs in subtle ways. Quick notes from memory:

- **Strava** — OAuth 2.0, scopes like `read,activity:read_all`. Tokens
  expire after 6 hours; refresh tokens are long-lived.
- **Polar AccessLink** — OAuth 2.0; uses HTTP Basic for the token
  exchange. Access tokens don't expire but require user registration
  via a separate POST after token exchange.
- **Wahoo Cloud API** — OAuth 2.0, access tokens expire after 2 hours,
  refresh tokens rotate on use.
- **Garmin Health/Activity API** — OAuth 1.0a (consumer key/secret +
  per-user oauth_token/oauth_token_secret). Different shape than the
  others; the callback receives an `oauth_token` + `oauth_verifier`
  rather than a `code`. Will sit alongside the existing `garth`-based
  Connect-web login, not replace it.

### `_AUTH_EXEMPT_ENDPOINTS` precedent

Adding new auth-related routes still requires appending to that set, as
called out in `DATABASE.md`'s "Common patterns and gotchas." Done for
these four. Future SSO additions: same drill.

### Preview deployments

The registered URLs only point at production
(`aidstation-pro.vercel.app`). Preview deployments get their own
`*-<hash>-andy-horns-projects.vercel.app` URLs and won't satisfy any of
the registered callbacks — OAuth flows can only be tested against prod
unless we register a wildcard or per-environment URI. Most providers
don't accept wildcards.

### Custom domain

If a custom domain (e.g. `aidstation.app`) is on the roadmap, register
*that* with the providers rather than the `*.vercel.app` URL. Provider
portals usually require re-submitting for review when the callback
changes, so getting the final domain right up front saves a round.

## Verifying the stubs

After Vercel finishes deploying `main`:

```bash
curl -i https://aidstation-pro.vercel.app/auth/strava/callback
# expect: HTTP/1.1 501 NOT IMPLEMENTED
#         Content-Type: text/plain; charset=utf-8
#         Strava OAuth callback not yet implemented.
```

Same shape for `/auth/garmin/callback`, `/auth/polar/callback`,
`/auth/wahoo/callback`. A `302` redirect to `/auth/login` means the
endpoint isn't in `_AUTH_EXEMPT_ENDPOINTS`; a `404` means the blueprint
isn't registered.
